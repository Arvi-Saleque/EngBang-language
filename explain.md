# How EngBang Compiles and Runs `my_test.txt`

This document traces the exact journey of the Student Score Analyser program through every stage of the EngBang compiler pipeline, pointing to the specific source files and functions involved at each step.

---

## The Pipeline (from `main.c`)

```
Source text
    │
    ▼
[1] Lexer       (lexer.l   → lex.yy.c)         tokenise the raw text
    │
    ▼
[2] Parser      (parser.y  → parser.tab.c)      check syntax, build AST
    │
    ▼
[3] AST         (ast.c / ast.h)                 tree of nodes in memory
    │
    ▼
[4] Semantic    (semantic.c)                    type-check, fill symtab
    │
    ▼
[5] TAC         (tac.c)                         emit Three-Address Code
    │
    ▼
[6] Execution   (exec.c)                        walk AST, produce output
    │
    ▼
[7] Cleanup     (ast.c, symtab.c, functab.c)    free all memory
```

`main.c` drives all stages in order:
```c
yyparse();            // stages 1–3
semantic_check(root); // stage 4
tac_generate(root);   // stage 5
exec_program(root);   // stage 6
ast_list_free(root);  // stage 7
```

---

## Stage 1 — Lexer (`src/lexer.l` → `src/lex.yy.c`)

**What it does:** Reads the source character by character and groups characters into *tokens* — the smallest meaningful units.

**How it starts:** `yyparse()` in the parser calls `yylex()` every time it needs the next token. `yylex()` is generated by Flex from `lexer.l`.

### What happens with our code

| Source text | Token returned | Rule in lexer.l |
|---|---|---|
| `//` | starts `LINECOMMENT` state — whole line discarded | `"//" { BEGIN(LINECOMMENT); }` |
| `50` | `INT(50)` | `{INTR} { yylval.ival = strtol(...); return INT; }` |
| `pakka` | `KW_PAKKA` | `"pakka" { return KW_PAKKA; }` |
| `rakho` | `KW_RAKHO` | `"rakho" { return KW_RAKHO; }` |
| `PASS_MARK` | `ID("PASS_MARK")` | `{IDR} { yylval.sval = dup_yytext(); return ID; }` |
| `->` | `ARROW` | `"->" { return ARROW; }` |
| `shonkha` | `TYPE_INT` | `"shonkha" { return TYPE_INT; }` |
| `\|` | `PIPE` | `"\|" { return PIPE; }` |
| `isPassing` | `ID("isPassing")` | `{IDR}` rule |
| `(` | `LPAREN` | `"(" { return LPAREN; }` |
| `>=` inside `[...]` | `COND_GE` | handled in `CONDGRP` state: `"{>=}" { return COND_GE; }` |
| `ferao` | `KW_FERAO` | `"ferao" { return KW_FERAO; }` |
| `barbe` | `KW_BARBE` | `"barbe" { return KW_BARBE; }` |
| `poriborton_by` | `KW_PORIBORTON_BY` | `"poriborton_by" { return KW_PORIBORTON_BY; }` |
| `thamo` | `KW_THAMO` | `"thamo" { return KW_THAMO; }` |
| `agao` | `KW_AGAO` | `"agao" { return KW_AGAO; }` |
| `3.0` | `FLOAT(3.0)` | `{FLOATR} { yylval.fval = strtod(...); return FLOAT; }` |
| `"hello"` | `STRING("hello")` | `{STR} { yylval.sval = dup_yytext(); return STRING; }` |

### Special: the CONDGRP state
When `[` is seen, the lexer switches to `CONDGRP` state. This allows `{>=}`, `{==}` etc. to be recognised as condition operators — they would not make sense as normal tokens. When `]` is seen, it switches back to `INITIAL`.

```
"["   → BEGIN(CONDGRP); return LBRACKET;
"]"   → BEGIN(INITIAL);  return RBRACKET;   (inside CONDGRP state)
```

### Memory: `dup_yytext()`
For identifiers and strings, the lexer calls `dup_yytext()` (defined at the top of `lexer.l`) which `malloc`s a copy of the matched text. This copy is passed as `yylval.sval` and eventually owned by the AST node.

## Stage 2 — Parser (`src/parser.y` → `src/parser.tab.c`)

**What it does:** Takes the stream of tokens from the lexer and checks that they form valid EngBang syntax according to the grammar rules. As it recognises each construct, it calls AST constructor functions to build a tree.

The parser is generated by **Bison** (LALR(1) shift-reduce parser).

### Grammar rules triggered by our code

**Constant declaration** — `50 pakka rakho PASS_MARK -> shonkha |`
```
const_decl → value KW_PAKKA KW_RAKHO ID ARROW type
           → ast_stmt_decl(id, type, expr, is_const=1, line)
```

**Function definition** — `isPassing(score -> shonkha) -> shonkha { ... }`
```
func_decl → ID LPAREN param_list RPAREN ARROW type LBRACE stmt_list RBRACE
          → ast_stmt_func_def(name, return_type, params, count, body, line)
```

**Conditional** — `score -> PASS_MARK { [{>=}] { ferao 1 | } }`
```
cond_chain  → ID ARROW value LBRACE cond_block_list RBRACE
cond_block  → LBRACKET group_list RBRACKET LBRACE stmt_list RBRACE
group       → LBRACKET cond_items RBRACKET   (the {>=} part)
```

**Loop** — `[score -> 40 -> shonkha, score -> 90] { ... score poriborton_by +15 }`
```
loop_stmt  → LBRACKET loop_init COMMA loop_end RBRACKET LBRACE stmt_list loop_ctrl RBRACE
loop_init  → ID ARROW value ARROW type           (declares new variable)
loop_ctrl  → ID KW_PORIBORTON_BY loop_step
loop_step  → PLUS INT    → { step_op='+', step_expr=ast_expr_int(15) }
```

**`agao` (continue)** — `agao |`
```
stmt → KW_AGAO PIPE  → ast_stmt_continue(line)
```

**`thamo` (break)** — `thamo |`
```
stmt → KW_THAMO PIPE  → ast_stmt_break(line)
```

**Cast declaration** — `gpTotal banaw doshomik rakho gpFloat -> doshomik`
```
cast_decl → value KW_BANAW type KW_RAKHO ID ARROW type
          → ast_stmt_cast_decl(id, cast_type, decl_type, value, line)
```

**When the grammar is violated**, Bison calls `yyerror()` which prints `PARSE_ERROR at line N: ...` and `yyparse()` returns non-zero — the pipeline stops in `main.c`.

### Result
After `yyparse()` returns, `g_parse_result` holds a **linked list of `AstNode*`** — the root of the entire program tree.

## Stage 3 — AST (`src/ast.c`, `include/ast.h`)

**What it does:** Provides the data structures and constructors that the parser uses. The AST is purely in-memory — there is no separate step; it is built *during* parsing.

### Node types used by our program

| EngBang construct | `StmtKind` | AST constructor |
|---|---|---|
| `50 pakka rakho PASS_MARK -> shonkha` | `STMT_CONST_DECL` | `ast_stmt_decl()` |
| `0 rakho passing -> shonkha` | `STMT_DECL` | `ast_stmt_decl()` |
| `khaali rakho x -> shonkha` | `STMT_DEFAULT_DECL` | `ast_stmt_default_decl()` |
| `totalScore + score bodlao totalScore` | `STMT_UPDATE` | `ast_stmt_update()` |
| `gpTotal banaw doshomik rakho gpFloat` | `STMT_CAST_DECL` | `ast_stmt_cast_decl()` |
| `bolo score` | `STMT_PRINT` | `ast_stmt_print()` |
| `score -> PASS_MARK { ... }` | `STMT_COND_CHAIN` | `ast_stmt_cond()` |
| `[score -> 40 -> shonkha, score -> 90] { ... }` | `STMT_LOOP` | `ast_stmt_loop()` |
| `isPassing(score -> shonkha) -> shonkha { }` | `STMT_FUNC_DEF` | `ast_stmt_func_def()` |
| `ferao total` | `STMT_RETURN` | `ast_stmt_return()` |
| `thamo` | `STMT_BREAK` | `ast_stmt_break()` |
| `agao` | `STMT_CONTINUE` | `ast_stmt_continue()` |

### Expression nodes

| Source | `ExprKind` | Constructor |
|---|---|---|
| `50` | `EXPR_INT_LIT` | `ast_expr_int(50, line)` |
| `3.0` | `EXPR_FLOAT_LIT` | `ast_expr_float(3.0, line)` |
| `score` | `EXPR_ID` | `ast_expr_id("score", line)` |
| `totalScore + score` | `EXPR_BINOP(BINOP_ADD)` | `ast_expr_binop(BINOP_ADD, left, right, line)` |
| `isPassing(score)` | `EXPR_CALL` | `ast_expr_call("isPassing", args, 1, line)` |

### How nodes link together
Each `AstNode` has a `next` pointer. `ast_list_concat()` chains them into a linked list representing the statement sequence. The loop `body`, function `body`, and conditional `block` are themselves linked lists stored as child pointers inside the parent node.

## Stage 4 — Semantic Analysis (`src/semantic.c`)

**What it does:** Walks every node in the AST, validates types, checks for undeclared variables, validates function calls, and registers all declarations in the **symbol table** (`symtab.c`).

### Two passes

**Pass 1 (prescan):** Registers all *function signatures* first so that forward calls work.
```c
g_prescan = 1;
sem_check_list(root);   // only handles STMT_FUNC_DEF → functab_register(...)
g_prescan = 0;
```
After pass 1, `isPassing`, `gradePoint`, and `sumFirst` are all in `functab`.

**Pass 2 (full check):** Walks every statement.

### What is validated in our program

**`50 pakka rakho PASS_MARK -> shonkha`**
→ `sem_check_decl()`: no initialiser type error, calls `symtab_declare("PASS_MARK", T_INT, is_const=1, ...)`.

**`isPassing(score -> shonkha) -> shonkha { ... }`**
→ `STMT_FUNC_DEF` case: calls `symtab_push_scope()`, declares param `score` as `T_INT`, recursively checks the body with `sem_check_list()`, then calls `symtab_pop_scope()`. Also tracks `g_current_ret_type = T_INT` so `ferao 1` is validated.

**`score -> PASS_MARK { [{>=}] { ferao 1 | } }`**
→ `sem_check_cond()`: looks up `score` in symtab, checks the binding expression `PASS_MARK` resolves to `T_INT` (matches), checks each group's condition count matches binding count.

**`[score -> 40 -> shonkha, score -> 90] { ... score poriborton_by +15 }`**
→ `STMT_LOOP` case:
- Calls `symtab_declare("score", T_INT, ...)` for the new loop variable.
- Checks init expr `40` is `T_INT` — matches.
- Checks end expr `90` is `T_INT` — matches.
- For `LOOP_CUSTOM`: checks step_expr `15` is `T_INT` — matches loop variable type.
- Recursively checks the body.

**`isPassing(score)` call inside the loop**
→ `EXPR_CALL` in `sem_check_expr()`: looks up `isPassing` in functab, checks argument count (1 == 1), checks argument type `T_INT` matches param type `T_INT`.

**`gpTotal banaw doshomik rakho gpFloat -> doshomik`**
→ `sem_check_cast()`: checks cast type == declared type (`T_FLOAT == T_FLOAT`), validates source expression, calls `symtab_declare("gpFloat", T_FLOAT, ...)`.

**`agao |` and `thamo |`**
→ `STMT_CONTINUE` / `STMT_BREAK` cases: both are always valid — just `break;` in the switch.

### Symbol table after semantic pass
`symtab.c` stores a flat linked list (`g_head`) of `Symbol` structs. After the full semantic pass, the global scope contains: `PASS_MARK`, `NUM_STUDENTS`, `passing`, `failing`, `totalScore`, `gpTotal`. Function-local variables are registered during semantic but then freed by `symtab_pop_scope()` — they are re-created at runtime by `exec.c`.

## Stage 5 — TAC Generation (`src/tac.c`)

**What it does:** Walks the AST and prints Three-Address Code (TAC) to stdout. This is for human-readable inspection — it does not affect execution.

### Key TAC patterns printed for our code

**Loop with `poriborton_by +15`:**
```
TAC: score = 40
TAC: t2 = 90
TAC: L18:
TAC: if score > t2 goto L19     ← ascending condition (+ operator)
TAC: ...body...
TAC: score = score + 15         ← custom step
TAC: goto L18
TAC: L19:
```

**`agao` (continue):**
```
TAC: continue
```

**`thamo` (break):**
```
TAC: break
```

**Function call `isPassing(score)`:**
```
TAC: arg score
TAC: t6 = call isPassing 1
```

**Cast:**
```
TAC: gpFloat = (doshomik) t18
```

TAC uses `new_temp()` for temporaries (`t0`, `t1`, ...) and `new_label()` for branch labels (`L0`, `L1`, ...). These counters are reset at the start of `tac_generate()`.

## Stage 6 — Execution (`src/exec.c`)

**What it does:** Walks the AST a second time, evaluating each node and updating the symbol table with runtime values.

### Key functions

| Function | Handles |
|---|---|
| `exec_program()` | Entry point — calls `exec_stmts(root)` |
| `exec_stmts()` | Iterates the linked list; stops if `g_returning`, `g_breaking`, or `g_continuing` is set |
| `exec_stmt()` | Dispatches on `n->kind` to the specific handler |
| `exec_decl()` | Calls `symtab_declare()` then `symtab_set_value()` |
| `exec_update()` | Calls `symtab_set_value()` |
| `exec_cast()` | Calls `cast_value()` then `symtab_declare()` + `symtab_set_value()` |
| `exec_print()` | Calls `exec_expr()`, prints based on type |
| `exec_cond_chain()` | Iterates blocks, calls `exec_cond_block()` |
| `exec_loop()` | Loop condition check + body + step advance |
| `exec_expr()` | Recursively evaluates an `AstExpr*` node into an `EvalVal` |

### Walking through our program

**`50 pakka rakho PASS_MARK -> shonkha`**
→ `exec_decl()`: calls `symtab_declare("PASS_MARK", T_INT, is_const=1, ...)`, then evaluates `INT(50)` → `EvalVal{T_INT, val.i=50}`, calls `symtab_set_value("PASS_MARK", T_INT, {.i=50})`.

**`isPassing(score)` call at runtime**
→ `exec_expr()` `EXPR_CALL` case:
1. Evaluates the argument `score` → `EvalVal{T_INT, val.i=40}`.
2. Calls `symtab_push_scope()` to isolate local variables.
3. Calls `symtab_declare("score", T_INT, ...)` and `symtab_set_value("score", ...)` with the argument value.
4. Saves and resets `g_returning` / `g_return_val`.
5. Calls `exec_stmts(f->body)` — runs the function body.
6. The `ferao 1 |` inside hits `STMT_RETURN`: sets `g_return_val = EvalVal{T_INT, val.i=1}` and `g_returning = 1`.
7. `exec_stmts` stops because `g_returning == 1`.
8. The return value is captured, `g_returning` is restored, `symtab_pop_scope()` frees `score`.
9. `EvalVal{T_INT, val.i=1}` is returned to the call site.

**Loop `[score -> 40 -> shonkha, score -> 90]` with `poriborton_by +15`**
→ `exec_loop()`:
- Declares `score` in symtab, sets it to `40`.
- Evaluates end expr `90` → `EvalVal{T_INT, val.i=90}`. Stored in `ev`.
- Each iteration: checks `score <= 90` (ascending because `step_op == '+'`).
- Body runs. If `agao |` is hit → `g_continuing = 1` → `exec_stmts` stops early for that iteration.
- After body: if `g_continuing == 1`, clears it and falls through to the advance step (correct — the step still runs).
- Advance: evaluates step_expr `15`, applies `nv.i += 15`.
- Loop ends when `score (100) > 90`.

**`agao |` inside the loop body (score == 70)**
→ `exec_stmt()` `STMT_CONTINUE` case: sets `g_continuing = 1`.
→ `exec_stmts()` sees `g_continuing == 1`, stops processing remaining statements in the body.
→ Back in `exec_loop()`: `g_continuing` is cleared, advance (`score += 15`) still runs → score becomes `85`.

**`thamo |` inside the countdown loop (c == 5)**
→ `exec_stmt()` `STMT_BREAK` case: sets `g_breaking = 1`.
→ `exec_stmts()` stops immediately.
→ Back in `exec_loop()`: `g_breaking == 1` → clears it, breaks out of the `for(;;)` loop entirely.

**`gpTotal banaw doshomik rakho gpFloat -> doshomik`**
→ `exec_cast()`:
1. Evaluates source `gpTotal` → `EvalVal{T_INT, val.i=5}`.
2. Calls `cast_value()` (`value.c`): converts `T_INT(5)` → `T_FLOAT(5.0)`.
3. Calls `symtab_declare("gpFloat", T_FLOAT, ...)`.
4. Calls `symtab_set_value("gpFloat", T_FLOAT, {.f=5.0})`.

**`gpFloat / count rakho avgGP -> doshomik`**
→ This is parsed as a normal `STMT_DECL` with value expression `gpFloat / count`.
→ `exec_expr()` `EXPR_BINOP(BINOP_DIV)`:
- Left: looks up `gpFloat` → `EvalVal{T_FLOAT, val.f=5.0}`.
- Right: looks up `count` → `EvalVal{T_FLOAT, val.f=3.0}`.
- Both float → result: `EvalVal{T_FLOAT, val.f=1.6667}`.

**`bolo avgGP |`**
→ `exec_print()`: evaluates `avgGP` → `T_FLOAT` → `printf("%g\n", 1.6667)` → prints `1.66667`.

## Stage 7 — Cleanup (`src/ast.c`, `src/symtab.c`, `src/functab.c`)

After execution:

- `ast_list_free(root)` — recursively walks every `AstNode` and `AstExpr`, calling `free()` on all owned strings and sub-trees. Every `malloc` from the lexer and AST constructors is paired with a `free` here.
- `symtab_free()` — frees the symbol linked list (`g_head` in `symtab.c`).
- `functab_free()` — frees function name strings and parameter arrays. (The function body AST is owned by the AST, not functab, so it is not freed here.)

## Full Data Flow Summary

```
"50 pakka rakho PASS_MARK -> shonkha |"
        │
        │  lexer.l
        ▼
  INT(50)  KW_PAKKA  KW_RAKHO  ID("PASS_MARK")  ARROW  TYPE_INT  PIPE
        │
        │  parser.y  → const_decl rule
        ▼
  ast_stmt_decl("PASS_MARK", T_INT, ast_expr_int(50), is_const=1, line=6)
        │
        │  ast.c → alloc_node(STMT_CONST_DECL)
        ▼
  AstNode { kind=STMT_CONST_DECL, decl.id="PASS_MARK",
            decl.declared_type=T_INT, decl.value=AstExpr{EXPR_INT_LIT,50},
            decl.is_const=1 }
        │
        │  semantic.c → sem_check_decl()
        ▼
  symtab: Symbol { name="PASS_MARK", type=T_INT, is_const=1, val.i=50 }
        │
        │  tac.c
        ▼
  TAC: PASS_MARK = 50
        │
        │  exec.c → exec_decl()
        ▼
  symtab_declare("PASS_MARK", T_INT, 1, 1, 6)
  symtab_set_value("PASS_MARK", T_INT, {.i=50})
  (runtime value stored — ready for use in isPassing)
```
